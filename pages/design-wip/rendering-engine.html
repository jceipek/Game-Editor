Rendering Engine

Coordinates

Dealing with pixels is annoying, especially given different aspect ratios and resolutions, so we are using a normalized coordinate system, as suggested in Jason Gregory's Game Engine Architecture.

There are 3 coordinate systems used in the engine:

Overlay

This coordinate style is used for absolute sprite positioning, such as when menus, health bars, or popups need to be displayed. In this system, the y position of overlays ranges from 0 at the top of the screen to 1.0 at the bottom of the screen. The range of the x coordinate is based on the y coordinate. For example, a 16:9 aspect ratio will yield 0-1.777.

Convention: When using variables with this coordinate style, prepend them with a lowercase 'o'. For example: oPosition
For Discussion: Gregory suggests using negative coordinates to refer to objects anchored at the lower left corner of the screen, which seems really clunky to me, especially since it offers no clean way to center something horizontally. How do we want to do this? - Julian
For Discussion: SFML uses degrees as the units of rotation. Do we want to use those for overlays and radians for physics rotations? - Julian

Physics

The game entities in the engine have positions in meters, because the physics engine is accurate at that scale. This system is used when you want to place or move characters or parts of the environment in a game.

Convention: When using variables with this coordinate style, prepend them with a lowercase 'm'. For example: mPosition

Pixels

The engine uses pixels internally when displaying graphics.

Convention: When using variables with this coordinate style, prepend them with 'px'. For example: pxPosition
Important: When you are making a game on top of the engine, you should never have to use this coordinate system.
